引擎识别这件事：现状、能力对照、与玩法的匹配方案
================================================================

一、当前状态：完全没有引擎识别这一层
----------------------------------------------------------------

回顾已经搭好的东西：
- src/sdk/GameSDK.ts、PlatformerSDK.ts —— 都是 Phaser 3
- docs/gdd-schema.md 不允许 meta.engine 字段
- Codegen prompt 强制只输出 config.json
- 测试只跑 Web (Playwright + Chromium)

也就是说：不管用户输入什么，目前 pipeline 100% 输出 Phaser 3。
这是 Level 0 阶段有意为之的设计选择 —— 先把 pipeline 跑通，再扩引擎。

为什么从 Phaser 起步：
1. 浏览器原生跑、无安装、最容易 headless 自动化测试
2. 纯 JS/TS，AI 出错后容易自动修
3. 部署成本零，分享一个 URL 就能玩

引入 Cocos / Unity 会带来三个量级完全不同的工程挑战：
- 构建管线（Cocos 项目结构、Unity 工程文件、UE 蓝图）
- 自动化测试（Cocos headless 较难，Unity Test Framework 需要 license + 复杂启动）
- 资产管线（场景、Prefab、AssetBundle 的格式）


二、如果要做"引擎识别"，正确的层级是什么
----------------------------------------------------------------

引擎选择不是单步推断，是一个多信号决策。
应该做成一个 Router Agent，输入是用户需求和能力清单，
输出是 (engine, genre, reasoning)。

决策信号（按权重）：

  信号                权重    来源
  --------------------------------------------------
  用户显式指定        最高    "用 Unity 做" → 直接选 Unity
  平台目标            高      "在浏览器玩" / "上 App Store" / "PS5 主机"
  性能/画质要求       中      "3D"/"AAA 画质"/"开放世界" → UE/Unity；
                              "轻量"/"H5" → Phaser/Cocos
  玩法类型            中      2D 横版 → Phaser/Cocos；3D 第一人称 → Unity/UE
  资产资源            中      用户已有 Unity assets → Unity
  团队/复用考虑       低      （生产环境才考虑）

决策表（草案）：

  3D 大型 / 开放世界 / 主机     → UE 5
  3D 移动 / 中型 / VR           → Unity
  2D 移动重交互 / 网游          → Cocos Creator
  2D Web / 一句话生成 Demo      → Phaser 3
  2D 像素 / 独立游戏            → Godot


三、引擎能力 × 玩法 的匹配矩阵
----------------------------------------------------------------

这是一份简化版能力对照。真实生产里这表会更细，这里给出思路。

引擎能力维度：
- 维度数：2D / 2.5D / 3D
- 物理：轻 (Phaser/Cocos) / 中 (Unity 内置) / 重 (UE/Havok)
- 网络：本地 / Photon (Unity) / Mirror / 自研
- 部署：Web / iOS / Android / PC / Console / VR
- 工具链：纯代码 / 编辑器 / 蓝图

引擎 vs 玩法 评分（5 = 极适合，1 = 不推荐）：

  玩法 \ 引擎       Phaser  Cocos   Godot   Unity   UE
  -----------------------------------------------------
  2D 跑酷            5      5       4       4       2
  三消               5      5       3       3       1
  平台跳跃           5      4       5       4       2
  塔防 (2D)          4      5       4       5       2
  卡牌 (2D)          5      5       3       4       2
  俯视角射击         4      4       4       5       3
  3D 第一人称        1      1       2       5       5
  3D 开放世界        0      0       2       4       5
  赛车               1      2       3       5       5
  MMO                0      1       1       4       5
  VR                 0      0       2       5       5
  轻量 H5 小游戏     5      4       3       2       1

简单说：
- Web 优先 + 2D                 → Phaser 3（当前选择）
- 2D + 复杂 UI / 网游           → Cocos
- 3D / 多平台 / 主流商业        → Unity
- AAA 画质 / 主机               → UE
- 像素风独立 / 开源              → Godot


四、做"引擎识别"的具体方案
----------------------------------------------------------------

方案 A：单引擎不动，先做"识别 + 拒答"（推荐起步）
- Schema 加 meta.engine，但只允许 phaser
- 加一个 Intent Agent，把用户需求里推出来的引擎和支持的对比
- 不匹配时降级 + 告知：
  "你想要 3D MMO，目前只支持 Phaser 2D，
   给你做一个最接近的 2D 太空跑酷，可以接受吗？"

方案 B：多引擎并行（投入大很多）
真要支持多引擎，需要给每个引擎复制一整套：
- 模板项目（Phaser 模板 / Cocos 模板 / Unity 模板）
- SDK 封装
- 自动化测试方案
- CI 沙盒

最现实的渐进路径：
1. Phaser（当前）
2. + Cocos Creator（同样 Web/Mobile 2D，移动友好）
3. + Godot（开源 + 跨平台 + 像素风社区）
4. （Unity / UE 暂时不做，自动化成本太高，个人玩不动）

每加一个引擎，至少要：
- 写一个该引擎的 minimal runner 模板
- 写一个 headless 测试
  （Cocos 用 Chromium 跑 web build；Godot 用 --headless + GDScript 自测）
- 在 codegen prompt 里加该引擎的 SDK 文档
- 在 router 里加规则


五、最小可落地变更（如要现在动手）
----------------------------------------------------------------

可以做这两步把"识别"框架先搭起来，但实际生成仍然只产 Phaser：

1. 加 Intent Agent 阶段
   在 codegen 之前，先用一个 prompt 推断
   (engine, genre, fallback_reason)

2. 加引擎能力清单文件
   docs/engine-capabilities.md，列出每个引擎能做和不能做的，
   作为 router 的知识源

3. 不匹配时返回兜底说明 + 降级到 Phaser

这样会立刻得到：
- 透明的引擎选择理由（每条评测都能看到 router 的判断）
- 为后续真的接入 Cocos / Godot 留好接口
- 没有破坏当前能跑通的 pipeline


六、最关键的一句
----------------------------------------------------------------

引擎选择不应该是模型一次"灵感判断"，
它应该是一个有规则、可解释、可降级的决策环节。

任何把 LLM 直接丢一句"该用什么引擎"的产品都会很快崩 ——
它今天选 Phaser，明天同样需求选 Unity，没有可重复性。
